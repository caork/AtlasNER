# 结构化抽取标注规范

**版本**: 1.0 | 2026-05-09

---

## 一、任务描述

给定一段电力领域文档文本，抽取其中的风险管控结构化记录。每条记录最多包含 6 个字段，仅输出文档中**确实存在**的字段。

---

## 二、字段定义

### 2.1 `equipment` — 设备

**定义**: 电力系统中被操作、检修、施工的具体设备、装置或部件名称。

**应标**:
- 一次设备: 变压器、断路器、隔离开关、GIS组合电器、电容器、电抗器、避雷器、互感器、母线
- 二次设备: 继电保护装置、自动化终端、通信设备
- 线路/通道: 架空输电线路、电缆、杆塔、基础
- 直流设备: 直流控制保护系统、换流阀、换流变压器

**不标**:
- 泛称: "设备"、"设施"、"系统"、"装置"（无具体指向时）
- 工具: 千斤顶、钢丝绳、绝缘绳（施工工具，非电力设备）
- 材料: 钢板、枕木、螺栓

**边界规则**:
- 电压等级 + 设备名 分开处理: `220kV变压器` → equipment="变压器"（电压在原始数据中通常不是设备字段的一部分）
- 设备 + 部件: `变压器高压套管` → equipment 取原文表述

---

### 2.2 `process` — 工序

**定义**: 电力系统中计划性的作业、操作、检修、施工流程或工作内容。

**应标**:
- 检修工序: 变压器整装转运、消防管道安装、回路绝缘检查
- 操作行为: 倒闸操作、分合闸、工作许可
- 施工流程: 基础开挖、铁塔组立、电缆敷设、架线
- 作业类型: 带电作业、停电检修、试验验收

**不标**:
- 事故/违章行为: 误操作（作为风险时归入 risk）
- 管理活动: 安全交底、班前会、方案审批

**与 `risk` 的区分**:
- "进行倒闸操作" → process（这是工序）
- "存在误操作风险" → risk（这是风险类型）
- 判断依据: 是否是计划内要执行的作业

---

### 2.3 `risk` — 风险描述

**定义**: 文档中描述的风险事件、事故类型、危险后果或风险点。

**应标**:
- 风险类型: 机械伤害、触电、高处坠落、物体打击、火灾爆炸
- 风险后果: 设备损坏、主变冲撞损坏、套管断裂、大面积停电
- 混合表述: "机械伤害、物体打击、主变冲撞损坏、套管断裂"（整体作为 risk 值）

**不标**:
- 纯工序描述（归入 process）
- 措施中提到的"防止XX"中的XX — 在 prevention 字段中保留原文即可

**格式**: 多个风险用原文分隔符（通常是顿号 `、` 或换行）。

---

### 2.4 `risk_level` — 风险等级

**定义**: 风险严重程度的等级标识。

**应标**:
- 文字等级: 高、中、低
- 完整表述: 重大风险、较大风险、一般风险、低风险
- 罗马数字: Ⅰ级、Ⅱ级、Ⅲ级、Ⅳ级

**不标**:
- "高风险作业"中的"高风险"（这是形容词修饰，非等级标识，除非文档明确将其作为等级分类）

---

### 2.5 `prevention` — 防范措施

**定义**: 文档中描述的风险防范措施、预控措施或安全措施。

**取值规则**:
- 保留完整的编号列表（`1. ... 2. ...`）
- 保留换行格式
- 不拆分、不精简、不合并
- 如果原文只有一条措施，也完整取出

---

### 2.6 `control` — 管控措施

**定义**: 工艺管控措施、质量管控措施或技术控制要求。

**与 `prevention` 的区分**:
- `prevention`: 侧重"防止风险发生"的安全措施
- `control`: 侧重"保证工艺质量"的技术措施

原文通常有明确的标题区分（"风险防范措施" vs "工艺管控措施"），按原文归类即可。

---

## 三、特殊情况处理

### 3.1 字段缺失

如果文档中**确实不存在**某字段的信息，JSON 中直接省略该键。不输出 `null`、空字符串或占位符。

### 3.2 多条记录

如果一个文档段落包含多个设备/工序的风险记录，输出 JSON array：

```json
[
  {"equipment": "变压器", "process": "整装转运", ...},
  {"equipment": "变压器", "process": "消防管道安装", ...}
]
```

当前数据中此情况罕见（每个 Excel 行是一条记录）。

### 3.3 无结构化信息的段落（负例）

如果文档段落是纯管理要求、制度说明等，不包含可提取的风险管控记录：

```
<think>
这段文档是一般性管理要求，不包含具体的设备、工序、风险或措施信息。无可提取的结构化记录。
</think>

```json
{}
```
```

### 3.4 措施引用标准

如果措施中引用了标准编号（如"按照Q/GDW 12152-2021要求"），标准编号保留在措施原文中，不单独提取为字段。

---

## 四、质量要求

### 4.1 Gold 数据

- Think chain 必须逻辑正确，字段定位依据明确
- JSON 字段值与文档原文一致（允许去除首尾空白和格式化差异）
- 措施字段条目数量完整（不遗漏）
- 缺失字段正确跳过（不臆造）

### 4.2 Inter-Annotator Agreement

Gold dev/test 数据建议双人审核，要求：
- 字段级 Cohen's Kappa ≥ 0.85
- 措施完整性分歧率 < 5%
